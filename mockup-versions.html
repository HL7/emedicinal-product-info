Managing the ePI Lifecycle

A structured approach to versioning, auditing, and traceability for ePI documents — from initial draft through regulatory approval, variation, and eventual withdrawal.

Version Identifiers at a Glance
Server-Managed
Bundle.id

A UUID that uniquely identifies the Bundle instance on the server. Not persistent across servers — will change if the ePI is copied or migrated.

Server-Managed
Bundle.meta.versionId

A version counter (1, 2, 3…) incremented automatically by the server with each update to this Bundle.

Business-Managed
Bundle.identifier

A UUID that stays constant across all versions of an ePI. Assigned by the authoring tool. Survives server migrations — used to group all versions of one ePI together.

Business-Managed
Composition.identifier + .version

A new UUID is generated for each document version to uniquely distinguish it. The version number (1, 2, 3…) is also incremented by the authoring tool.

Composition.relatesTo — Relationship Types
ValueWhen to Use
derived-from Creating a new draft by copying content from an existing final ePI (e.g., starting a variation or safety update).
amends A new approved ePI modifies or adds content to a previous approved ePI — used in merge scenarios where parallel variations are consolidated.
replaces A newer version (draft or final) completely supersedes an earlier version within the same ePI lineage.
retracts Formally withdraws or invalidates a previous version — used with status = deprecated or entered-in-error.
Version Control — Step by Step
1
Creating a New Version
  • New Bundle created with a new Bundle.id and meta.versionId = "1"
  • Bundle.identifier stays the same as previous versions
  • New Composition.identifier and incremented .version
  • Composition.relatesTo references the prior version
2
Updating an Existing Version
  • Server increments Bundle.meta.versionId and updates meta.lastUpdated
  • Bundle.identifier remains unchanged
  • New Composition.identifier and .version generated if this constitutes a new document version
3
Tracking & Audit
  • Both Bundle.identifier and Composition.identifier allow tracking across systems
  • Composition.relatesTo builds a version history chain
  • Server-managed fields ensure FHIR versioning compliance
